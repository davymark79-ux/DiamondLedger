# Diamond Ledger — Baseball Sim

A from-scratch baseball simulation engine + React UI ("Diamond Ledger"). Plain JS (not TypeScript), Vite + React + Tailwind, `oxlint` for linting. Now a git repository (initialized partway through the build) — `git log` has real history from that point forward, but this file and `diamond-ledger-design-docs/` remain the primary record for design intent and the reasoning behind decisions.

**Design docs**: `diamond-ledger-design-docs/` — `README.md` is the index. These are the source of truth for *intended* mechanics. The actual code sometimes diverges from them in deliberate, flagged ways (documented inline in the relevant files and summarized below) — usually because a doc leaves an algorithm/exact-value unspecced, or depends on a system that doesn't exist yet.

**Running it**: `npm run dev` (Vite dev server). No automated test framework — verification is via the `npm run validate:*` scripts (see below) plus manual browser checks.

## Architecture / layering convention

- **`src/models/`** — schema + generation only. No simulation logic.
  - `Player.js`, `Team.js`, `League.js`, `constants.js` — core schema. Ratings are 20-80 scale, three-layer (`current` / `truePotential` / `scoutedPotential`).
  - `generation/` — seeded generators. `playerGenerator.js` has **two distinct modes**: `generatePlayer()` builds HS/pre-draft prospects (age 17-18, only 30-70% of true potential realized — raw, unfinished talent, position picked from a weighted pool) and `generateEstablishedPlayer()` builds active-roster-ready players (age 21-37, 85-100% of true potential realized, **requires an explicit `position`** — no random draw, since real rosters need guaranteed position coverage). `random.js` (`createRng`, `randomInRange`, `pickWeighted`, `pick`, `gaussianRandom`) is a deterministic mulberry32 PRNG — `rng` is **always** an explicit passed-in parameter throughout the whole codebase, never an implicit `Math.random()`.
  - `seed/` — `leagueSeed.js` (the real, locked-in 50-team structure transcribed from `league-structure.md`: 30 MLB1 teams in 3 divisions + 20 MLB2 teams in 2 divisions, each division mixing Foundry-league and Exchange-league teams roughly half-and-half), `rosterSeed.js` (attaches real 26-man rosters to those teams).
- **`src/engine/`** — simulation mechanics that read/transform Player/Team objects. Deliberately non-mutating (`growthModel.js`'s explicit "returns a new object" convention) — the one exception is `engine/season.js`'s external accumulator `Map`s (standings, rotation index, injury status), which are caller-owned state, not mutations of the Player/Team objects themselves.
- **`src/data/`** — UI-facing, eagerly-computed data. `realLeague.js` and `season.js` are the *real* pipeline: both computed **once at module load with a fixed seed** (not `Date.now()`) — deliberately a stable, consistent world across the whole app, not regenerated per page visit. `hallOfFame.js` is a third, similar static-seed precedent, but sourced from a one-time-run precomputed snapshot script rather than computed at module load (see section 18). `gameContext.js` (venue/attendance/game-time flavor for `BoxScore.jsx`, see section 5) is the one small `Date.now()`-seeded exception, and only for that one cosmetic derivation — the rosters/teams it's given are always the real singleton's. `mockData.js` (old placeholder data that pre-dated the real engine) is **gone** — every page now reads real data (see sections 5, 19); nothing in `src/` imports it anymore.
- **`src/pages/`** — route components.

## Build order / current state

### 1. Schema — done
`Player` / `Team` / `League` + `constants.js`. Attribute groups: `HITTING_ATTRIBUTES`, `BASERUNNING_ATTRIBUTES`, `DEFENSE_ATTRIBUTES`, `PITCHING_ATTRIBUTES` (pitchers only), `MAKEUP_ATTRIBUTES` (`workEthic`, `durability`, `consistency`, `coachability`). `Player.injury` — `null` when healthy, otherwise `{type, severity, gamesRemaining, sustainedGameNumber}`. Note: this field documents the *shape*, but the **authoritative** injury tracking lives externally in `engine/season.js`'s `injuryStatusById` Map, not by mutating this field on the shared `realLeague.js` Player objects (keeps that singleton pure).

### 2. Plate-appearance engine — done
`plateAppearance.js` + `plateAppearanceConstants.js`, validated via `npm run validate:sim` against real modern-MLB per-PA rate stats (`BASE_RATES`, `BASE_BABIP` are anchored to ~.248/.312/.410-ish league averages). **Do not inflate these constants** to fix an aggregate run-environment problem elsewhere in the pipeline — tried once, it corrupted the anchored neutral-matchup baseline; the real fix belonged elsewhere (see Calibration notes).

### 3. Full game loop — done
`game.js` orchestrates: baserunning (`baserunning.js`, includes double plays and sac flies), pitcher fatigue/times-through-the-order/bullpen management (`pitcherFatigue.js` + `timesThroughOrder.js` → `pitcherDegradation.js`; `pitchingChanges.js` decides pulls and next-pitcher), position-player substitutions (`hitterChanges.js` — pinch-hit / defensive-replacement / pinch-run, each now with a no-quality-gate `selectEmergency*` variant for injuries), per-game "form" consistency (`consistency.js` — `applyGamePerformance` / `applyPitcherGamePerformance` roll once per player per game; `withPerformanceModifiers` is the general "shift these ratings by a delta" primitive, reused by injuries and reserved for a future "playing hurt" mechanic), the ghost-runner extra-innings rule (`gameRules.js`, default off), box-score tracking (`boxScore.js` — batting/pitching lines, GIDP, RISP), pitcher decisions (`pitcherDecisions.js` — Win/Loss/Save/Hold/Blown-Save; a save requires the lead was *never* relinquished during the outing, not just checked at entry/exit), injury rolls (`injuries.js`, hooked into the per-PA/per-pitch loop), and position-player fatigue (`positionPlayerFatigue.js` — a per-game performance dip applied in `createSide()`, cross-game tracking owned by `engine/season.js`).

**Hard invariant, keep this pattern for any future substitution type**: `removedPlayerIds` (a `Set` per side) — once a player is subbed out for any reason, he can never be selected again. Enforced both structurally (spliced out of the pool array) *and* defensively (an explicit Set check inside every selector function). Stress-tested at scale (hundreds of games) before being trusted.

### 4. Growth Model + position reassignment — done
`growthModel.js` — period-over-period career development (`growth = baseline_growth + variance_roll`), calibrated so most players don't reach true potential. Extension-point hooks `locationModifier` (future college-prestige system) and `injuryImpact` (available for a future *development-period* injury penalty — **not** currently wired to the Injuries system that got built, which only handles in-game/season availability, not growth-period impact).
`positionReassignment.js` — fit-driven + aging-driven triggers only; org-depth and injury-driven triggers are still explicitly deferred (need Team-level roster depth charts / deeper Injuries integration than exists).
`reassignmentReaction.js` — Coachability-gated accept/refuse mechanic.
`development.js` composes all three. Validated via `npm run validate:growth`.

### 5. Box Score UI — done, real rosters
`BoxScore.jsx` lets the user pick any two of the real 50 teams (two `<select>` dropdowns, each excluding whichever team is picked in the other, defaulting to two random distinct teams on load) and simulates a real game between their **real** rosters — `data/season.js`'s `buildRealMatchup(awayTeamId, homeTeamId, rng)` reuses `engine/season.js`'s own `buildGameSide`/`resolveAvailableRoster`/`resolveRestedRoster` (exported for exactly this reuse, the same functions the season loop itself calls per game), so this is subject to the *live singleton's current end-of-season state* — a player currently on the shelf in `injuryStatusById` or fatigued in `consecutiveGamesPlayedById` is correctly unavailable/rested here too, and each team's real current manager's sliders apply. Each team starts `rotation[0]` (its de facto ace) — a standalone game has no rotation-index to cycle, unlike the season loop. "Simulate Again" re-runs the same two teams with a fresh rng; "Random Matchup" rerolls both teams. Shows a full real box score: stacked decision badges (W/L/S/H/BS — a pitcher can carry more than one, e.g. blown-save+win is a real combo), GIDP, substitution notes (including `injury`-type ones), lineup indentation for subs under whoever they replaced, and same-surname name disambiguation.

`src/data/gameContext.js` (replacing the deleted `simDemoRoster.js`, which used to *generate* a synthetic matchup — no longer needed now that real rosters exist) derives venue/attendance/game-time flavor from the real matchup's `marketSize` and the actual simulated pitch count — the one remaining `Date.now()`-seeded piece, and only for this cosmetic layer.

**Not real yet, a deliberate scope line**: the roster/player *generation* itself is unaffected by this — no new career-simulation or Growth-Model integration was added, this only swapped which rosters a one-off game draws from (synthetic → the real, already-built 50-team league).

### 6. Real roster construction — done
`realLeague.js` + `rosterSeed.js`: all 50 real teams have a full, real **26-man roster** (9 lineup + 5 rotation + 8 bullpen [4 named roles: Long/Middle/Setup/Closer, + 4 unnamed depth arms] + 4 bench), generated via `generateEstablishedPlayer()`. MLB1 vs. MLB2 quality bands are deliberately ~20 points apart (`ROSTER_QUALITY_BY_TIER`); bench/bullpen-depth players get a further `DEPTH_QUALITY_DISCOUNT` below their team's starters (a real downgrade, not a lateral move). A small free-agent pool also exists (`freeAgents`, `teamId: null`) but nothing consumes it yet. `getTeamRoster(teamId)` resolves a roster's id array into `{lineup, rotation, bullpen, bench}` — **`bullpen` is ordered `[Long, Middle, Setup, ...4 depth arms, Closer]`**, Closer deliberately kept last because `selectNextPitcher`'s save-situation logic treats the last array slot as the closer.

Historical note: this roster originally generated only 18 of 26 spots (8 reserved for a planned HS draft). That reservation was abandoned once the Injuries system needed real bench/bullpen depth to substitute into — see the Injuries section. The *original* broader intent (fill each org's entire minor-league pyramid, not just the MLB active roster) was never built and still isn't — no AAA/AA/A rosters exist anywhere.

### 7. Season/schedule loop — done
`engine/season.js` + `data/season.js`. Schedule generation groups teams by `(tier, leagueId)` — **no interleague regular season** (`league-structure.md`: Foundry only plays Foundry, Exchange only plays Exchange, until the Cup/Finals) — and builds ~150 games/team via repeated shuffled round-robin cycles (the exact pairing *algorithm* isn't specced anywhere, this is a placeholder; the 150-games target and no-interleague rule *are* specced). `simulateSeason()` runs every scheduled game through the real engine: DH-aware lineups (Foundry = no DH, that day's starter bats; Exchange = DH), 5-man rotation cycling, real win/loss standings, and injury persistence (below). `Standings.jsx` / `Schedule.jsx` / `TeamDetail.jsx` all read this real data now. `Overview.jsx` / `Cup.jsx` / `Events.jsx` / `Financials.jsx` still read `mockData.js`'s old fictional placeholder data — untouched.

Not modeled: playoffs, the Cup tournament, promotion/relegation *execution* (the real rule — a 1-for-1 swap, each league's last-place MLB1 club trades places with that league's first-place MLB2 club — is documented in `Standings.jsx`'s copy but nothing acts on it), roster expansion, the draft, free agency, real calendar/dates (games are an ordered sequence, not tied to actual dates — `season-calendar.md`'s full preseason/blackout-weekend/All-Star/playoff sequencing is a separate, later step).

### 8. Injuries and injury substitutions — done
`engine/injuries.js` implements `injuries.md`'s risk formula: `base_rate × position_multiplier × age_multiplier × durability_modifier × fatigue_multiplier[pitchers only]` (the fatigue term reuses `pitcherFatigue.js`'s existing `computeFatiguePenalty()` rather than reinventing pitcher tiredness). This is `durability`'s **first real mechanical use** anywhere in the codebase.

Severity tiers (`DAY_TO_DAY` / `SHORT_TERM_IL` / `LONG_TERM_IL` / `SEASON_ENDING` / `CAREER_ENDING`) carry **hard floors** — `gamesRemaining` never reaches zero before the real-IL-rule minimum elapses (10-day/60-day equivalents), regardless of how "healed" a player narratively is. A setback mechanic layers real recovery variance on top (a short-term injury can escalate to a more severe tier) — **bounded to a short window right after diagnosis/escalation, not a flat ongoing per-game hazard**, because a flat hazard compounds catastrophically with a long tier's duration (see Calibration notes — this was a real bug, caught and fixed).

In-game, only the **current batter** (after his PA resolves) and the **current pitcher** (after each pitch) roll for injury — not the other 7 fielders or non-batting baserunners, since this engine has no fielding-event or baserunning-event attribution to hook a risk into. On a hit, it forces a real substitution (pitcher side reuses the already-unconditional `selectNextPitcher`; hitter side uses new `selectEmergency*` variants in `hitterChanges.js` — same eligibility logic as the tactical selectors, minus the "must be a clear upgrade" quality gate). If nobody's available, the player continues — the same graceful-degradation fallback the engine already uses everywhere else a substitution pool runs dry.

`engine/season.js` persists injuries across games via `injuryStatusById` (same external-Map ownership pattern as `standingsById`) and filters injured players out of each game's roster (promoting an available bench player into the slot). `TeamDetail.jsx` shows `IL · Ng` / `OUT` tags on injured players; `BoxScore.jsx` shows `Injury: X for Y` substitution notes. Validated via `npm run validate:injuries`.

**Explicitly flagged, real future work — not generic background noise, specifically asked to be remembered**:
- **Fielding injuries** — needs a real fielding-event mechanic first (doesn't exist; defense currently has zero effect on outcomes at all).
- **"Playing hurt"** — every triggered injury is an immediate full substitution today; there's no "grits through a nagging injury at reduced performance" state. `consistency.js`'s `withPerformanceModifiers()` is the natural primitive to reuse once this is built.

Also deferred: cascading minor-league call-ups (blocked on the unbuilt Minor League System), the full AAA/AA/A organizational depth chart, real calendar-date-based healing that survives season boundaries (blocked on the still-nonexistent calendar model — durations are tracked in games missed, not real dates), catcher-specific injury-type severity profiles, CBA-negotiable IL duration.

### 9. Position-player fatigue — done
`engine/positionPlayerFatigue.js` implements `injuries.md`'s "Position Player Fatigue" section — a deliberately much lighter-touch companion to pitcher fatigue: `computeFatiguePenalty(consecutiveGamesPlayed)` (0 until a 6-game rest threshold, then climbs, capped at 5 — vs. pitcher fatigue's 28-point cap) and `computeFatigueRiskMultiplier(...)` (feeds into `injuries.js`'s `computeInjuryRisk` as the non-pitcher `fatigue_multiplier` term, scaled smaller than the pitcher version too). `applyFatigue()` nudges every on-field physical attribute (`HITTING_ATTRIBUTES`+`BASERUNNING_ATTRIBUTES`+`DEFENSE_ATTRIBUTES`) down by a flat amount — deliberately *not* an independent per-attribute roll like `consistency.js`'s form modifier, since this is a workload penalty, not a talent-consistency swing.

Cross-game tracking is owned by `engine/season.js`'s `consecutiveGamesPlayedById` Map (same ownership pattern as `injuryStatusById`) via `advanceFatigueForTeam()`, called once per side after each simulated game: a player who started gets his counter incremented, anyone who didn't (benched, hurt, rested — see below — or, for a Foundry no-DH team, the nominal DH-slot roster player, who doesn't actually play that game at all) gets reset to 0. `TeamDetail.jsx` shows a `Fatigued · Ng` tag (distinct color from the `IL` tag) next to affected position players — pitchers are never tracked by this mechanic at all (their own heavier fatigue system + the 5-day rotation cycle already serve as rest). Validated via `npm run validate:fatigue`.

**Auto-rest (added once the Manager entity existed, see section 16)**: `positionPlayerFatigue.js`'s `computeRestProbability(consecutiveGamesPlayed, analyticsVsFeel)`/`rollRest(...)` let a manager proactively bench a fatigued player before a game, driven by the **Analytics vs. Feel** slider — the doc's "data-driven models vs. gut instinct" framing maps directly onto the real-world load-management archetype (an Analytics manager proactively rests regulars; a Feel manager just keeps running them out). Reuses `computeFatiguePenalty`/`MAX_FATIGUE_PENALTY` (now exported) as the fatigue-severity fraction rather than a second "games over" constant. **Zero at/below a neutral (50) slider value** — deliberately asymmetric unlike the other five sliders, since there's no *prior* auto-rest behavior to preserve at neutral (nothing ever auto-rested before this); only `analyticsVsFeel > 50` starts proactively managing workload, so every caller without a real manager (every validate script, the standalone demo) sees zero behavior change. `engine/season.js`'s `resolveRestedRoster(roster, consecutiveGamesPlayedById, managerProfile, rng)` is a structural twin of the existing `resolveAvailableRoster` — applied *after* it in `simulateSeason`'s per-game loop (injuries are mandatory, rest is a voluntary managerial choice layered on top), same "first unused bench player in array order" simplification `resolveAvailableRoster` already uses (no position-eligibility check there either). `advanceFatigueForTeam`'s existing "not in playedIds → reset to 0" logic needed zero changes — a rested player already flows through the exact same code path as a benched/injured player. Confirmed live: a real team under a manager with `analyticsVsFeel` just barely above neutral (51/100) already shows meaningfully lower fatigue counts across its whole roster than before this change (previously several `Fatigued · 154g` maxed-out tags were common; now none). Validated via `validate:fatigue`'s extended sections 5-7, including a real-season comparison (an Analytics-leaning manager's regulars topped out at 9 consecutive starts vs. a neutral team's 154 — the cap-plateau problem this was built to fix, closed).

### 10. Hot/Cold Streaks — done
`engine/hotColdStreaks.js` implements `player-attributes-and-development.md`'s "Hot/Cold Streaks" section — purely descriptive, does **not** feed back into performance (respected by `plateAppearance.js`'s own header comment). Built as the first step toward the full Manager entity (see "Path to Managers" below): `Player.js`'s `streakState` field and `constants.js`'s `HOT_COLD_TIERS`/`HOT_COLD_THRESHOLDS` (±0.75/±2.0 SD five-tier table) already existed but were never computed anywhere until now.

The metric generalizes the doc's own worked example (a .250 hitter's expected 5-for-20, binomial SD ≈1.94) from a flat batting-average rate to a weighted, wOBA-like composite (`WOBA_WEIGHTS`, real-MLB-anchored linear weights): a batter's context-neutral baseline mean/variance is computed via `plateAppearance.js`'s extracted `computeOutcomeProbabilities(batter, pitcher)` (a pure refactor — `resolvePlateAppearance()` now just rolls against that same distribution) against a synthetic `LEAGUE_AVERAGE_PITCHER`, not his actual opponents — matching how real rate-stat projections are contextualized.

**Window shape — an exponentially-decaying (EWMA) accumulator, not a fixed game count.** A flat, uncapped cumulative sum was tried first and rejected: once a real hot stretch accumulates a large excess, a flat sum barely forgets it, requiring an unrealistic pile of contrary evidence (~1,000+ PA) to revert — nowhere close to "cool off within a month or so." `updateStreakState()` instead tracks a decaying `{effectivePa, effectiveTotal}` pair (`EWMA_DECAY_PER_GAME = 0.92`, half-life ≈8-9 games): a real ongoing streak keeps reading as a streak for as long as it's actually sustained (no fixed cutoff), while a genuine reversal fades back to Neutral within a believable few-week span. Below `MIN_STREAK_SAMPLE_PA` (16) effective PA, a reading is unconfirmed — `tier` reports `NEUTRAL` regardless of how extreme the raw SD looks, so a two-game burst (e.g. 4-for-4 then 1-for-4) never counts as a real streak.

Cross-game tracking is owned by `engine/season.js`'s `streakAccumulatorById` (internal, decaying accumulator per player) and `streakStateById` (the latest computed reading, returned from `simulateSeason()`) — same external-Map ownership pattern as the other cross-game accumulators. `TeamDetail.jsx` shows a `Cold`/`Ice Cold`/`Hot`/`On Fire` tag (gold for Hot/On Fire, blue for Cold/Ice Cold) **only for a confirmed streak** — the gating above means a one-game blip never surfaces a tag. Validated via `npm run validate:streaks`.

### 11. Platoon/handedness splits — done
`Player.js`'s `bats`/`throws` fields — purely cosmetic until now — genuinely affect outcomes via `plateAppearance.js`'s `platoonShift(batter, pitcher)`. A rating-point-equivalent shift per handedness pairing (`plateAppearanceConstants.js`'s `PLATOON_SHIFTS`), added into the existing skill-diff calculations for strikeout/walk/home-run/BABIP (sign flips for strikeout specifically, since that diff is oriented pitcher-minus-batter, opposite of the other three — documented inline, guarded by `validate:sim`'s directional asserts). Deliberately asymmetric per real MLB data: LHB vs LHP is the single toughest pairing, a bigger penalty than RHB vs RHP — not just a flat same-hand/opposite-hand split. Switch-hitters always resolve to the side opposite the pitcher's throwing hand (`effectiveBattingSide()`), always taking the platoon advantage.

**New rated attribute — `platoonSkill`** (`MAKEUP_ATTRIBUTES`, 20-80 scale): individual players vary in how much the population-level shift applies to them. `scaleFactor = 1 - (platoonSkill - 50) * PLATOON_SKILL_SCALE_SENSITIVITY` — 1 (unchanged) at the population average, compressing toward 0 as it climbs (a flatter-split "matchup-proof" hitter), growing above 1 as it falls (a classic short-side platoon player), and — per an explicit user ask — crossing slightly negative only at the true tail of the scale (~80, already rare by this engine's population-scarcity design), a small genuine reverse-split. Being a normal rated attribute (not a fixed-by-birth constant) means it's generated, scouted, and developed period-over-period with **zero extra wiring** — `createPlayer()`/`playerGenerator.js`/`growthModel.js` all already treat `MAKEUP_ATTRIBUTES` generically.

A pitcher with no assigned `throws` (e.g. `hotColdStreaks.js`'s context-neutral `LEAGUE_AVERAGE_PITCHER` reference, which predates this feature) is treated as handedness-neutral — zero shift, not a crash or `NaN` — the semantically correct reading for a reference opponent that shouldn't carry a specific platoon edge either way.

**Real calibration wrinkle, worth remembering**: `createPlayer()` defaults `bats`/`throws` to `'R'`, so any synthetic fixture that doesn't explicitly randomize handedness is an implicit same-handed (worst-case) matchup. `validate-game-loop.mjs`'s 500-game aggregate hit exactly this — its fixtures had always been implicitly all-same-handed (harmless before this feature existed), and once the platoon penalty was real, avg runs/game dropped from ~8.24 to ~6.71, a real regression, not an expected side effect. Fixed at the root cause: `makeBatter`/`makePitcher` there now assign handedness via the real generator's own `pickBats`/`pickThrows` (newly exported from `playerGenerator.js` for exactly this reuse) instead of leaving it defaulted, restoring ~8.28/8.53. **Lesson**: any synthetic engine fixture that doesn't explicitly set `bats`/`throws` is implicitly same-handed and will read as a systematically harder matchup than a realistic mixed-handedness roster — reuse `pickBats`/`pickThrows` rather than leaving handedness at the default when building a new fixture meant to represent a realistic population.

Also fixed in the same pass: `engine/season.js`'s `advanceFatigueForTeam()` had a one-game lag — a player injured *during* a game still got his fatigue counter incremented for that same game, since the pre-game resolved roster couldn't know he'd get hurt mid-game. Now also checks `injuryStatusById` (already updated with this game's injuries by the time fatigue advances) and force-resets immediately rather than waiting a game. Pre-existing edge case, not caused by platoon splits — just surfaced by the new RNG-driven outcome sequence exercising it for the first time.

Validated via the extended `npm run validate:sim` — new platoon-pairing rows, directional/magnitude/switch-hitter asserts, a population-weighted-average anchor check (using the generator's real `pickBats`/`pickThrows` proportions), and `platoonSkill` scaling asserts (compress/exaggerate/rare-flip). No lineup-construction or pinch-hit decision layer was built — this unlocks the *mechanic* the future Platoon Tendency manager slider will act on, not the manager decision itself.

### 12. Stolen bases — done
`engine/stolenBases.js` — nothing about stealing existed before this: no attempt decision, no success resolution, no box-score fields. Not specced in any design doc (only `managers.md` mentions a future Steal Aggressiveness manager slider); this engine's own placeholder logic, same status as `pitchingChanges.js`/`hitterChanges.js`'s own un-specced substitution decisions.

**Scope, confirmed with the user**: only 1st→2nd and 2nd→3rd (steals of home deferred — rare, situational, mostly suicide-squeeze territory in reality). Because a runner can only attempt when the very next base is open, `identifyStealOpportunity(bases)` finds **at most one** opportunity per plate appearance — real double steals (multiple runners breaking simultaneously) aren't modeled; a runner on first is simply blocked if second is occupied, but the runner on second in that same state still has his own independent shot at third if it's open.

`computeStealAttemptProbability(runner, situation, scoreMargin)` — a runner's own Speed/Baserunning-Instincts composite (reusing `hitterChanges.js`'s `speedComposite()`, now exported, the same "runner quality" concept already used for pinch-run decisions) shifts a per-situation base rate; zero outright once `|scoreMargin|` exceeds a threshold (not worth the risk in a blowout either way). **Deliberately has no innings-from-end gate** unlike the three `hitterChanges.js` substitution mechanics — stealing is a general in-game tactic, not an endgame-specific one.

`computeStealSuccessRate(runner, catcher, pitcher, situation)` — runner composite vs. the catcher's Arm Strength/Accuracy composite (their **first real gameplay mechanical use** anywhere — previously only fed `positionReassignment.js`'s fit-scoring weights), plus a smaller pitcher `control` term as a rough stand-in for "hold time/quickness to the plate" (an explicit user choice — no dedicated attribute exists for this, and control was judged a reasonable existing proxy over inventing a new one). `FIRST_TO_SECOND` and `SECOND_TO_THIRD` have distinct baseline attempt/success rates (illustrative placeholders, matching this project's tuning-status tone) — 2nd→3rd is rarer-attempted and a bit riskier.

**Wired into `game.js`'s per-half-inning loop**: `maybeAttemptSteal()` runs **before** `nextBatter(offense)` — critical ordering, since `nextBatter` mutates `offense.battingIndex` as a side effect. A caught-stealing increments `pitcherLine.outsRecorded` (a real out, affects fatigue/TTOP) but deliberately **not** `battersFaced` (the same batter is still up). If a caught-stealing is the 3rd out, the existing `while (outs < 3)` loop naturally handles it correctly — **the batter at the plate never gets a PA that half-inning**, matching real baseball (he leads off next inning). Exported for direct testability (matches `season.js`'s `resolveAvailableRoster` precedent) since this correctness property was judged important enough to unit-test in isolation rather than only trust by inspection.

`boxScore.js` gained `sb`/`cs` battingLine fields + `recordStolenBaseAttempt()`; `BoxScore.jsx`'s `BattingTable` gained SB/CS columns (same tabular treatment as the existing 2B/3B/HR/BB/K columns). Validated via `npm run validate:steals` — attempt/success direction, situation-baseline differences, the mid-PA-ending-inning correctness property (unit-tested directly against `maybeAttemptSteal` with a fixed-sequence fake `rng`), `outsRecorded`/`battersFaced` correctness, and a real-season wiring check (~0.5 SB/team/game in a 100-game smoke test). Confirmed live: SB renders correctly on `/box-score`.

### 13. Bunting / Small Ball — done
`engine/bunting.js` — nothing about bunting existed before this. Not specced in any design doc (only `managers.md`'s Small-Ball Tendency slider mention); this engine's own placeholder logic, same status as `stolenBases.js`. **Scope, confirmed with the user**: includes the suicide/safety squeeze (bunting with a runner on 3rd to score him) as one higher-drama "runner unconditionally breaks" mode — no separate lower-risk safety-squeeze variant (no Manager entity exists yet to drive that finer distinction).

**Key architectural difference from stolen bases**: a bunt is an *alternative resolution of the same at-bat*, not an independent mid-PA event — so it reuses the exact same `recordPlateAppearance()` call site already in `game.js`'s loop. `maybeAttemptBunt()` returns either `null` (proceed with a normal `resolvePlateAppearance`/`resolveBaserunning` pair, unchanged) or a `resolveBaserunning`-shaped result (`outcome` forced to `PA_OUTCOMES.OUT`) — everything downstream (pitch count, injury checks, box-score recording, run-scoring) stays a single unconditional call site either way. No early-`break` needed (unlike stolen bases) since a bunt attempt *is* the batter's full turn.

**New rated attribute — `buntingSkill`** (`HITTING_ATTRIBUTES`, not `MAKEUP_ATTRIBUTES` like `platoonSkill`): bunting is a physical/technical bat-control skill, so it *should* get the same per-game form jitter and position-player-fatigue nudge contact/power/eye already get. `buntComposite(player)` blends `buntingSkill` (0.7) and `speed` (0.3) — real speed/bunting correlation, per the user's explicit ask, modeled at the formula level rather than correlating the two attributes' generation (a bigger, riskier change to `playerGenerator.js`'s uniform per-attribute sweep — a deliberate scoping choice, not an oversight).

`computeBuntAttemptProbability()` layers several real factors the user specifically asked to have reflected, on top of a per-situation base rate (`SQUEEZE` rarer than `PLAIN`): a Foundry (no-DH) bonus (this league's established small-ball identity), a larger batting-pitcher bonus (only ever fires in a Foundry game anyway, so it naturally stacks), and a late-game/runner-on-first-only/0-1-out "manufacture a run" bonus (matching `hitterChanges.js`'s existing innings-from-end convention). `computeBuntSuccessRate()` is `buntComposite`-driven only — no defensive factor (predates fielding/positioning below, which was still deferred at the time this shipped).

**New export in `baserunning.js`** — `resolveSacrificeBunt(bases, batter)` reuses the existing private `advanceOnHit(bases, batter, 1)` helper (handles bases-loaded force-in-a-run correctly already) and discards the batter's own placement — a clean reuse of an existing primitive rather than new advancement logic.

**Plumbing addition**: `dhRule` now rides along on each side object (`season.js`'s `buildGameSide()` → `game.js`'s `createSide()`, same pattern as `consecutiveGamesPlayedById`), defaulting to `true`/Exchange (no bonus) for callers that don't know/care, like the standalone `/box-score` demo.

`boxScore.js` gained `sh` + extended `recordPlateAppearance()` (sacrifice credit only on success, matching real MLB rules — a failed attempt is a normal recorded AB/out); `BoxScore.jsx` gained an SH column. Validated via `npm run validate:bunt` — situation classification, attempt/success direction, the Foundry/batting-pitcher/manufacture-a-run bonuses, and forced success/failure on both `PLAIN` and `SQUEEZE` (including the critical double-out risk: a failed squeeze costs both the batter *and* the runner from third, unlike a failed plain sacrifice). Confirmed live on `/box-score`.

### 14. Fielding / Positioning (Defensive Management) — done
The biggest gap closed this session: **defense had zero mechanical effect on any outcome since the very first session** (`plateAppearance.js`/`baserunning.js`'s own long-standing header notes). `engine/fielding.js` — not specced in any design doc (only `managers.md`'s Defensive Management slider mention). This is Fielding/Arm Strength/Arm Accuracy's **first real effect on any outcome anywhere** — previously they only fed `positionReassignment.js`'s fit-scoring weights.

**Also fixed, a separate pre-existing gap found during investigation**: a runner on third **never scored on a groundout**, ever, regardless of positioning — `baserunning.js`'s groundball branch only ever handled the runner-on-first case. Confirmed with the user to fix this too (it's the prerequisite for modeling "infield in" at all).

`computeTeamDefenseComposite(defense)` averages Fielding (0.6)/Arm Strength (0.2)/Arm Accuracy (0.2) across the 8 real fielders currently on the field (excludes `DH` and the pitcher). Wired into `plateAppearance.js`'s `computeOutcomeProbabilities`/`resolvePlateAppearance` via a new optional 3rd param `{composite, noDoublesActive}` (safe defaults preserve every existing caller — `hotColdStreaks.js`'s baseline, `statLine.js`'s isolated matchup simulator — unchanged) — **BABIP/hit-split only**, deliberately; defense doesn't affect strikeouts, walks, or home runs in reality either.

**Three situational positioning tactics**, matching the slider's own description:
- **No-doubles** (`isNoDoublesActive` — defense leading, last 2 innings) shifts hit-type split toward singles, away from doubles/triples.
- **Double-play depth** turned out to be a direct extension of `baserunning.js`'s *existing* `rollDoublePlayChance` formula (which previously only read batter speed) — not a new decision layer at all, just a `defenseComposite` term added to it.
- **Infield-in** (`isInfieldInActive` — defense tied or down modestly, last 2 innings) is layered onto the newly-fixed groundout-scoring logic: on a successful throw home, the runner from third is out at the plate instead and the *batter* reaches first safely (the real trade-off, a different out at the cost of a new baserunner) — on failure, same as the routine case (run scores, batter out at first).

**Errors**: a ball that would otherwise be `PA_OUTCOMES.OUT` can instead become a fielding error (`computeErrorChance`, worse defense commits more), checked in `game.js` *before* `resolveBaserunning` runs (an error preempts whatever groundball/flyball/DP/infield-in sub-resolution that out would have gone through). New `baserunning.js` export `resolveReachedOnError` reuses `advanceOnHit(bases, batter, 1)` directly — unlike `resolveSacrificeBunt`, the batter *does* reach base this time, no override. Team-level `side.errors` counter (no per-player attribution — this engine has no specific-fielder-per-play resolution to support that). **Simplified same-play-only unearned-run rule**: no RBI and no earned run credited when a run scores on the same play as an error (a real MLB scoring convention); a runner who reached on an *earlier* error but scores later via a separate clean play is still counted earned in this v1 model — an explicit, flagged simplification short of full inning-reconstruction logic.

**Real bug caught by the validation script, not glossed over**: the first `rollDoublePlayChance` extension had `defenseComposite`'s term *subtracted* (copy-pasted the batter-speed term's sign without flipping it) — good defense was actually turning *fewer* double plays. `validate:fielding`'s directional assert caught it immediately; fixed to add the term instead.

`BoxScore.jsx`'s `LineScore` gained an E column next to the existing R column. Validated via `npm run validate:fielding` (11 sections — the most thorough script this session, matching the biggest blast radius: core PA engine *and* core baserunning touched simultaneously) — composite/BABIP/error/no-doubles/infield-in/DP-chance direction, all four situational trigger conditions, the groundout-scoring fix directly, `resolveReachedOnError`, `recordPlateAppearance`'s `isError` handling, and real-season wiring. Re-ran the full regression suite afterward — `validate:game`'s runs/game actually moved *up* toward the ~8.6-9.0 target (previously suppressed by the missing groundout-scoring case, now fixed) rather than down like the last two additions — a good sign the fix was a real, overdue correction, not just added noise. Confirmed live on `/box-score`: E column renders correctly.

### 15. Origin/Nationality/Retirement (cosmetic-only pass) — done
`managers.md` needs two things from `Player` before manager generation can reuse them: a nationality tag system ("ex-player managers draw nationality directly from their existing player record") and player retirement to exist at all ("an ex-player manager has to have stopped playing first"). Both were designed-but-unbuilt: `Player.js` already had `birthNation`/`heritageNation` fields, but every generated player was hardcoded `birthNation: 'USA'`, `heritageNation: null`; retirement didn't exist anywhere (no field, no function, no removal logic).

**Nationality** (`international-tournament-and-nationality.md`'s "shallow data, rich mechanic" design): new `models/generation/nationalityPools.js` — `BIRTH_NATION_POOL` (weighted, ~70% USA / ~30% foreign-born, illustrative placeholder matching real MLB demographic proportions) and `HERITAGE_NATION_POOL` (the birth pool plus several heritage-only nations) feed `pickBirthNation`/`pickHeritageNations`, wired into both `generatePlayer()` and `generateEstablishedPlayer()`. `heritageNations` (renamed from the unused singular `heritageNation`) holds 0-2 entries per `HERITAGE_NATION_COUNT_WEIGHTS` (78/18/4%, matching the doc's 75-80/15-20/2-5% split). **Explicit scope-down from the doc**: no regional weighting (e.g. "Miami-born skews Cuban") — no birth-region/city data exists on `Player` to drive that correlation, so heritage is drawn independently of birth nation instead. Also explicitly deferred: localized name pools by nationality (`namePools.js` stays USA-only regardless of `birthNation`). `TeamDetail.jsx` gained a `NationalityTag`, parallel to `InjuryTag`/`FatigueTag`/`StreakTag` — deliberately quiet: a plain-USA-no-heritage player (the common case) shows nothing, anyone with a non-USA birth nation or any heritage tag gets a small code (e.g. `DOM`, `VEN`) with full detail on hover.

**Retirement** (`engine/retirement.js`, new): `managers.md` originally sketched a bare age/decline soft-probability default. **Real user feedback reshaped this significantly** — retirement should be rare mid-season and mostly an end-of-season decision, and a season's injury outcome should be a real input, not just age/decline. Three triggers, evaluated together in one end-of-season roll per player:
1. Age/decline voluntary retirement — soft probability curve (0 below 32, rising through the high-30s/40s, no hard cutoff, matching Writers Corps'/Scouts'/Managers' own soft-retirement shape). Above age 33, a decline term adds further probability, approximating "decline below career peak" via current-vs-true-potential (no real peak-tracking field exists on `Player` — an explicitly flagged approximation, since a bust who never developed reads similarly to a declined veteran under this proxy).
2. Injury-driven, reusing `injuries.js`'s existing `INJURY_SEVERITIES` rather than inventing new state: `CAREER_ENDING` forces retirement outright (probability 1); `SEASON_ENDING` adds a real probability bump, but **only** for a player already at/past the same age-33 gate — "an older player who maybe planned to retire at season's end gets a season-ending injury and just calls it there," deliberately not applied to a young player with the same injury (he's expected back).
3. DFA'd-and-not-picked-up — **explicitly deferred, not built**, per direct user instruction to leave it for a future master roster-transaction system rather than a loose proxy now. No DFA/waiver/roster-transaction mechanic exists anywhere in this codebase (no trades, no waivers, no roster moves at all). Documented as a specific, named extension point in `engine/retirement.js`'s own header comment — the same "computed hook, not yet acted on" pattern already used for `reassignmentReaction.js`'s `requestsTrade`/`benchWorthy` flags.

`advanceCareerForRoster(roster, rng, {asOfDate, injuryStatusById})` rolls retirement across `lineup`/`rotation`/`bullpen`/`bench`, returns `{roster, retiredPlayerIds}` with retirees filtered out — same shape/precedent as `season.js`'s `resolveAvailableRoster()`, non-mutating like `growthModel.js`.

**Real constraint surfaced and accepted, not routed around**: this app has no season-to-season progression loop at all — `season.js`'s `simulateSeason()` plays one fixed schedule once, and `growthModel.js`/`development.js` (period-over-period aging) are themselves never called from that loop either, despite being fully built and validated. Retirement follows the exact same precedent: `advanceCareerForRoster` is real, tested, and ready for a future season-advancement driver to call at a season boundary, but isn't wired into `simulateSeason()` this pass (there's no boundary to hook it to, and the mechanic is meant to fire there, not mid-game anyway). No `RetiredTag` was added to `TeamDetail.jsx` for the same reason — nothing in today's live singleton roster data will ever actually retire.

Validated via `npm run validate:origin` (`scripts/validate-origin-and-retirement.mjs`) — nationality distribution/dedup checks, retirement probability direction (age, decline term, both age-gated), `CAREER_ENDING` forcing probability to exactly 1, a young player's `SEASON_ENDING` injury NOT bumping his probability, `rollRetirement`/`advanceCareerForRoster` correctness. All checks passed first run. Re-ran the full regression suite (touches `Player.js`'s schema and every generated player's `birthNation`) — all pass, no script asserted on the old `birthNation`/`heritageNation` shape. Confirmed live on `/teams/:id`: non-USA players show a nation code, USA-born players with heritage tags show `USA` with heritage detail on hover, plain-USA players show nothing.

`managers.md` (v0.4) assembled in full: schema, generation, per-team assignment, and every one of the seven sliders wired into a real decision — the final item on the roadmap below.

High Temperament → tight noise; low → wide.

A **streak bonus** only applies when Analytics-vs-Feel is Feel-leaning (an Analytics manager ignores streak state entirely, per the doc) and runs the player's real Hot/Cold tier through `perceiveStreakTier` before applying it — a Feel-leaning manager with poor Streak Read can genuinely misjudge who's actually hot. ### 17. Manager Firing & Rehiring — done
`managers.md`'s Career Lifecycle ties Firing to "the main design doc's existing 'Coaches fired' Scripted Event category (triggered by win%, chemistry, owner patience)." That entry (`baseball-sim-design-doc.md`) turned out to be the *entire* spec — three named factors, zero numeric thresholds, no generic Scripted Event framework ever built (only disconnected mock flavor data existed). Of the three factors: **win%** was real but not exposed as a reusable helper; **owner patience** had zero elaboration anywhere in the docs; **chemistry** has a genuine future design in `player-relationships.md` ("computed as an aggregate... of the current roster's individual Org Affinity scores"), itself unbuilt. Built the trigger from **win% + a new, honestly-flagged owner-patience stat** — chemistry stays explicitly deferred with its real, named blocker, not faked or glossed over.

**A good realization during design**: unlike player/manager retirement (which need a season-*to*-season boundary that doesn't exist), firing is naturally an *in-season* event — `mockData.js`'s own old flavor text already assumed this ("Manager dismissed after 14-32 stretch"). This is wired directly into `engine/season.js`'s existing single-season loop, not left standalone-and-unwired like the two retirement mechanics.

`engine/managerFiring.js` (new): `computeWinPct(wins, losses)`; `updateOwnerPatience(currentPatience, won)` — a bounded 0-100 random walk, asymmetric on purpose (a loss drains it faster than a win restores it — "owners get impatient faster than grateful"); `computeFiringProbability(winPct, gamesUnderManager, ownerPatience)` — 0 below a minimum sample (`MIN_GAMES_BEFORE_FIRING_ELIGIBLE`, 20 games *under the current manager*) and 0 at/above .500 (no team fires a .500-or-better skipper under this model); patience *amplifies* the win%-driven probability rather than acting independently (a bad record under a patient owner is safer than the same record under an impatient one).

**Tenure, not season-to-date**: `season.js` gained `tenureRecordById` (wins/losses under the *current* manager, reset on every hire) — deliberately not the team's whole-season `standingsById` record, so a new hire gets a genuinely clean evaluation window instead of inheriting his predecessor's bad record and risking an almost-immediate re-firing.

**The real "career ladder" the doc wants**: `availableManagerPool` (starts empty) — a fired manager is pushed there (excluded from immediately replacing himself, since the replacement pick happens before the push); a new hire pulls from the pool first, only generating a fresh manager when it's empty. `managerAssignmentById` (pre-seeded from `getTeamManager` at loop start) is the only thing every per-game manager lookup reads from then on, since assignments can change mid-loop; a `firings` log (`{gameNumber, teamId, firedManagerId, hiredManagerId, winPctAtFiring}`) records every event.

**Backward compatibility, same pattern as auto-rest**: the whole mechanic is gated on a team actually having a real manager assigned (`getTeamManager` returning non-null) — a team running on the neutral synthetic default (every existing validate script that doesn't pass `getTeamManager`) is never fired or hired, zero behavior change. Confirmed by re-running `validate:season` unchanged. Production (`data/season.js`) passes a real `getTeamManager` for all 50 teams, so Firing & Rehiring is fully live there — 25 real firings occurred league-wide over a full 150-game season in testing, and pool-reuse (a fired manager hired by a different club) is *deterministic*, not just probable, once 2+ firings happen anywhere in the league (the pool starts empty, so only the very first firing ever generates a fresh manager — every firing after that necessarily finds a non-empty pool).

`data/season.js` gained `getCurrentTeamManager(teamId)` (the season's final assignment, not the static seed one) and `getTeamManagerChanges(teamId)`; `TeamDetail.jsx`'s `ManagerCard` switched to the season-aware getter and gained a compact "Manager Changes" note when a team's manager changed in-season. Confirmed live: New York's `ManagerCard` correctly shows its new manager (hired from Camden, NJ) with "Fired after game 1504 (.481)" for the predecessor.

Validated via the extended `npm run validate:managers` (sections 14-18) — `computeWinPct`/`computeFiringProbability`/`updateOwnerPatience` direction and gating, the deterministic pool-reuse invariant, and real-season wiring. All checks passed first run. Re-ran the full 13-script suite — all pass, `validate:season` (no `getTeamManager` passed) completely unaffected.

### 18. Hall of Fame — done

**Season-total stat accumulation** (`engine/season.js`): `seasonBattingStatsById`/`seasonPitchingStatsById`, summed from each game's box-score lines plus W/L/S. `BATTING_STAT_FIELDS`/`PITCHING_STAT_FIELDS`/`emptySeasonBattingTotals`/`emptySeasonPitchingTotals`/`groupTeamsForScheduling` were promoted from module-private to exported for reuse by the files below.

**`engine/hallOfFame.js`** — `isHallEligible` (the two LOCKED numbers, ported verbatim from real BBWAA rules: 10 years service, 5 years retired); `computeBatterHallCaseScore`/`computePitcherHallCaseScore`/`computeManagerHallCaseScore` (weighted counting-stat sums). **Two real bugs caught and fixed while building `validate:hof`/the snapshot script**:
1. `validate:hof`'s section 5 forced retirement deterministically via a constant `rng = () => 0` — but `advanceOffseason` runs growth first, which calls `gaussianRandom(rng, ...)` internally, and `gaussianRandom` has `while (u === 0) u = rng()`, which spins forever against a true constant-zero rng. Fixed by using `() => 0.0001` instead. **A constant-zero rng is safe for a plain `rng() < probability` comparison but not safe as a general "force everything" rng anywhere `gaussianRandom` gets touched** (growth variance, scout-error rolls, Temperament/Streak-Read noise, etc.) — use a small nonzero constant there instead.
2. `seasonsPlayedById` was incrementing once per player per *stat source* (once for appearing in `seasonBattingStatsById`, again for `seasonPitchingStatsById`) rather than once per season — a Foundry (no-DH) pitcher, who both pitches and bats, was accumulating two seasons of "service" per real season played. Caught via a sanity print, not `validate:hof` itself (a real gap in that script's coverage — it checked accumulation and plausibility but never an explicit `seasonsPlayed <= totalSeasonsSimulated` upper bound, which would have caught this directly). Fixed by unioning both key-sets into one `Set` before incrementing.

**A real calibration bug, also caught by a dry run, not a formula-shape unit test (those can't catch a magnitude-scale mismatch against real simulated data)**: the snapshot script's first draft used a shortened 100-game season to save build time; a 20-season trial's best career topped out at case score 38.6, nowhere near `hallOfFame.js`'s 70-point vote-center — meaning zero candidates would ever get elected, no matter how good, because the case-score formula's milestone anchors are calibrated to real 150-game-season career pacing and a shortened season structurally can't produce milestone-scale counting stats. Fixed by switching to the real 150-game length (confirmed: a 10-team/15-season/150-game trial reached case score 93.1) — timing math confirmed this isn't meaningfully slower (~1279 games/sec, full run ~100s). **When a formula is anchored to real-world numbers, always sanity-check it against the actual data the engine produces before trusting the formula in isolation** — a shortened-for-speed simulation parameter can silently break the anchor's whole premise.

Validated via `npm run validate:hof` (8 sections). Ran the snapshot script for real: 25 simulated seasons, 773 candidates reached eligibility within the window, 87 inductees (85 regular ballot + 2 via Legacy Committee), 169 still active on the ballot at snapshot end — all case scores comfortably above the 70-point vote center, no duplicate ids. Re-ran the full 14-script regression suite afterward (`sim`/`game`/`growth`/`roster`/`season`/`injuries`/`fatigue`/`streaks`/`steals`/`bunt`/`fielding`/`origin`/`managers`/`hof`) — all pass. ## Key conventions

- **Every numeric constant is an explicit, commented placeholder** ("needs real playtesting") — a deliberate tone across the whole engine, not sloppiness. Don't hesitate to add more of these; don't try to "finalize" a number without being asked to.
- **`rng` is always an explicit parameter**, never an implicit `Math.random()`.
- **One-concern-per-file**, composed by a thin orchestration layer (e.g. `pitcherFatigue.js` + `timesThroughOrder.js` → `pitcherDegradation.js`; `growthModel.js` + `positionReassignment.js` + `reassignmentReaction.js` → `development.js`).
- **"Never returns" / one-time-use invariants** (bullpen, bench, injured-player exclusion) are enforced both structurally *and* defensively. Stress-test any new invariant like this at real scale before trusting it.
- **Extension-point pattern** for systems that don't exist yet, rather than building them prematurely (`locationModifier` / `injuryImpact` on `growthModel.js`).
- **Tailwind gotcha, hit twice already**: `` `grid-cols-[${var}]` `` template-literal interpolation breaks Tailwind's static class scanner — the *complete* class string must appear literally in the source file (a `const X = 'grid-cols-[...]'` declaration is fine and gets picked up; interpolating a variable *into* the brackets at the point of use is not). When in doubt, hardcode the literal class string at every JSX use site, matching `BoxScore.jsx`'s verbose-but-safe pattern.
- **`EnterPlanMode`** for any genuinely new system with real design decisions to negotiate (used for every major build-order step above); skip it for well-precedented extensions of an already-approved pattern.
- **Trace calibration drift to its root cause** before touching a shared/anchored constant — see below.

## Calibration notes — read before touching any of these

Several systems were jointly tuned and need re-validation together if any one changes:

- `plateAppearanceConstants.js` (real-MLB-anchored — **do not** inflate to fix an aggregate run-environment issue elsewhere), `pitcherFatigue.js`, `pitchingChanges.js`, `timesThroughOrder.js` — target ~8.6-9.0 runs/game, ~4.3-4.6 runs/team/game, but only in `validate:game`'s **symmetric average-vs-average** matchups.
- Real teams (`realLeague.js`) have wide, independently-randomized per-player quality — mismatched-quality games naturally run lower-scoring than the symmetric baseline (~7.5-8.5 runs/game in `validate:season`, a *loose* sanity bound, not the tight symmetric-matchup target). This is expected, not a bug — don't chase it by inflating PA constants.
- `validate-game-loop.mjs`'s own synthetic bullpen fixture uses a realistic role-based quality curve (`BULLPEN_QUALITY` — Long/Middle weakest, Closer best), not a flat quality bullpen — this was the actual fix for an earlier "saves are too rare" investigation. Don't revert it.
- Starter pitch counts should land around 85-107 (stamina-dependent), not the 120-146 an earlier tuning pass allowed. If they creep up again, fix `pitchingChanges.js`'s pull thresholds directly — don't loosen them back up just to hit a run-environment number; find that fix elsewhere (see the bullpen-quality-curve precedent above).
- Injury setback rate: **never** use a flat per-game-check hazard for an injury's *entire* duration — it compounds explosively with tier length (a 3%/check hazard over a 60-game floor is ~84% cumulative, not "some players have setbacks"). Bound the window (`SETBACK_WINDOW_GAMES`) instead.
- `createPlayer()` defaults `bats`/`throws` to `'R'` — any synthetic engine fixture that doesn't explicitly randomize handedness (via `playerGenerator.js`'s exported `pickBats`/`pickThrows`) is an implicit same-handed matchup, which now reads as a systematically *harder* matchup than a realistic mixed-handedness roster since platoon splits exist. Caught this exact issue in `validate-game-loop.mjs` (see Platoon splits section) — check any *new* synthetic fixture for this too.
- Stolen bases and bunting both trade a small amount of run environment for realism — caught-stealing removes baserunners, sacrifice bunts trade a guaranteed out for one base (a real, lower-expected-value play in most situations, matching real sabermetric consensus). `validate:game`'s runs/game has drifted modestly downward across both additions (~8.24/8.46 baseline → ~7.94/8.32 after bunting) — **expected**, not a regression to chase, unlike the handedness-fixture bug above. The distinguishing question when a number moves: is a fixture behaving unrealistically (root-cause and fix it), or is a real mechanic legitimately consuming outs/bases it didn't before (document it and move on)?
- Fielding/positioning reversed that trend: `validate:game`'s runs/game rose from ~7.94/8.32 to ~8.96/8.79. Not a regression — the groundout-scoring fix (`baserunning.js`'s `resolveOut`) closed a real gap where a runner on third could **never** score on a routine groundout, in any prior session. That was suppressing runs every session up to this one; fixing it was always going to raise the number, and it landed back inside the ~8.6-9.0 reference band rather than outside it. Team-defense-driven BABIP/error/no-doubles/infield-in effects partially offset it but don't reverse the direction. Confirmed via `validate:fielding`'s own directional asserts, not just the aggregate move.
